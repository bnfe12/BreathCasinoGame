# Design: AI Defense + Deck Lifecycle + Gun Return
**Date:** 2026-03-27
**Status:** Approved
**Approach:** Targeted changes, no new classes

---

## 1. Scope

Four fixes in three existing files:

| File | Change |
|---|---|
| `EnemyAI.cs` | AI defense tries 2-card combination if no single card covers attack |
| `BlockoutCardManager.cs` | Weight 8 forbidden for Resource cards; new `RecreateCurrentRoundDeck()` method; auto-recreate when deck exhausted |
| `GameManager.cs` | Save gun original position at round start; restore it after shot |

## 2. EnemyAI — `ChooseDefenseCards()`

### Current behavior
Picks the minimum-weight single Resource card with weight >= attackWeight.
If none found → returns empty list (skip).

### New behavior
1. Try single card (same as now).
2. If not found → try all 2-card combinations of Resource cards.
3. Among valid pairs (sum >= attackWeight) pick the one with minimum sum.
4. If no valid pair → return empty list (skip).

### New helper method
```csharp
private List<BlockoutCardDisplay> FindBestDefenseCombination(
    IReadOnlyList<BlockoutCardDisplay> availableCards,
    int attackWeight)
```
- Iterates all pairs, tracks minimum-sum pair that covers attackWeight.
- Returns list of 2 cards or empty list.
- Never includes Threat cards (same guard as existing code).

### Constraint
Maximum 2 cards in defense (slot limit from MECHANICS.md).

## 3. BlockoutCardManager

### 3.1 Weight 8 forbidden for Resource

In `CreateWeightedMainCard()`:
- If `!forceThreat` and the sampled weight == 8 → re-clamp to 7.
- Threat cards (`forceThreat == true`) retain weight 8.
- Threat cards generated by the inline check (`isThreat` flag from weight+random) use `{4, 6, 8}` — unchanged for Threat, but the Resource path is clamped to max 7.

Concretely:
```csharp
if (!forceThreat)
    weight = Mathf.Clamp(weight, minWeight, Mathf.Min(maxWeight, 7));
```

### 3.2 `RecreateCurrentRoundDeck()` — new public method

New private field `_lastKnownRoundIndex` (set in `EnsureDeckForRound` each call):
```csharp
private int _lastKnownRoundIndex = 0;
```

```csharp
public void RecreateCurrentRoundDeck()
{
    int roundToRecreate = _lastKnownRoundIndex;
    _mainDeck.Clear();
    _mainDiscard.Clear();
    _currentDeckRoundIndex = -1;           // force EnsureDeckForRound to rebuild
    EnsureDeckForRound(roundToRecreate);
    Debug.Log($"[CardManager] Deck recreated for round {roundToRecreate + 1}");
}
```

- Stores `_lastKnownRoundIndex` (new private field, set in `EnsureDeckForRound`).
- HP is NOT touched — only deck state resets.
- Does NOT call `ClearAllHands()` — players keep their current cards.

### 3.3 Auto-recreate in `RefillHandsToRoundLimit()`

After standard refill loop, add:
```csharp
if (_mainDeck.Count == 0 && _mainDiscard.Count == 0)
{
    RecreateCurrentRoundDeck();
    // Retry refill (only once)
    RefillMainHand(...);
    if (allowSpecials) RefillSpecialHand(...);
}
```

**Order of events:**
1. Try standard refill (draws from deck, reshuffles discard if needed — existing behavior).
2. If both deck and discard empty → `RecreateCurrentRoundDeck()`.
3. Refill again from fresh deck.

## 4. GameManager — Gun Position & Visual

### Problem A: gun doesn't appear "in hands"
`MoveGunToHolder()` already re-parents gunRoot to `playerWeaponHolder`, but that transform
may not be positioned in front of the FPS camera.
Fix: ensure `playerWeaponHolder` is parented to the camera (or camera rig) at a sensible
FPS offset (e.g. `localPosition = (0.15, -0.15, 0.35)`).
This is a **scene wiring fix**, not a code change — set it in the Inspector on the
`playerWeaponHolder` transform so gunRoot lands in view when picked up.

### Problem B: gun drifts / doesn't return to exact spot
`MoveGunToTable()` already does `SetParent(gunSpot, false); localPos = zero; localRot = identity`
which is correct. The visual "not returning" is likely caused by `UpdateGunAim()` running
every frame while `_waitingForPlayerShot = false` is already set — LookAt keeps rotating
the gun even after the shot resolves.

Fix: add a guard in `UpdateGunAim()`:
```csharp
private void UpdateGunAim()
{
    if (!_waitingForPlayerShot) return; // already have this
    ...
}
```
(Already present — confirm it is hit correctly after shot.)
The real fix is to call `MoveGunToTable()` **after** `_waitingForPlayerShot = false` is set,
not before, so `UpdateGunAim()` stops before the table-move runs.

### New field: reset flag on full restart
```csharp
private bool _gunReturnedToTable; // set true after MoveGunToTable(), false in MoveGunToHolder()
```
Reset `_gunReturnedToTable = true` in `BeginTestGame()` so a full game restart
doesn't leave stale gun state.

### `MoveGunToTable()` — explicit reset
```csharp
private void MoveGunToTable()
{
    if (_scene?.gunRoot == null || _scene?.gunSpot == null) return;
    _scene.gunRoot.SetParent(_scene.gunSpot, false);
    _scene.gunRoot.localPosition = Vector3.zero;
    _scene.gunRoot.localRotation = Quaternion.identity;
    _gunReturnedToTable = true;
}
```

### `MoveGunToHolder()` — mark gun as "in hands"
```csharp
private void MoveGunToHolder(BlockoutSide side)
{
    ...existing logic...
    _gunReturnedToTable = false;
}
```

### After shot
`ResolveShotCoroutine()` already calls `MoveGunToTable()` at the end.
Order is: set `_waitingForPlayerShot = false` → … → `MoveGunToTable()`.
`UpdateGunAim()` checks `_waitingForPlayerShot` first, so rotation stops before table-move.

## 6. Risk Assessment

| Area | Risk | Mitigation |
|---|---|---|
| AI 2-card combination | Could make AI too hard in round 1 with few cards | Pair search only when single card fails — same fallback (skip) |
| Deck recreate | Could flood round 1 with too many cards | Deck sizes unchanged (12/17/24); only triggers when both deck+discard empty |
| Gun position | gunSpot might not be set in scene | Guard `_scene?.gunSpot != null` already present |
| Weight 8 clamp | Could break Threat card generation | Clamp only applied when `!forceThreat` |
